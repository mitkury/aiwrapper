# Real-time AI Agent Reference Guide based on LiveKit JS/TS

## Overview
This proposal outlines the core components and architecture of the LiveKit Agents-JS package that can be referenced or adapted for building a real-time AI agent capable of handling voice, video, and general multimodal interactions. LiveKit provides a robust, production-ready framework that manages complex real-time operations, and examining its design can greatly accelerate our own agent development.

## 1. Core Agent Pipeline Architecture (`VoiceAgent` / `Agent`)
The LiveKit framework structures its voice agents around a pipeline of interconnected nodes. We can adopt a similar pipeline architecture:

*   **Node-Based Processing (`voice/agent.ts`):** LiveKit breaks down the agent's work into distinct nodes:
    *   `sttNode`: Speech-to-Text for transcribing user audio.
    *   `llmNode`: Large Language Model for generating responses based on context.
    *   `ttsNode`: Text-to-Speech for converting the LLM's text back into audio.
    *   `realtimeAudioOutputNode`: Handling the final audio output to the user.
*   **Agent Configuration (`AgentOptions`):** Agents are configured with specific instructions (system prompts), tools (functions they can call), and model selections for STT, LLM, and TTS.
*   **Activity Tracking (`AgentActivity`):** LiveKit manages the current state of the agent's tasks, preventing overlapping execution and handling speech queues.

**What we can use:** The concept of a modular, stream-based pipeline where audio flows into STT, text flows into LLM, and text flows into TTS, all while maintaining a centralized context.

## 2. Session Management (`AgentSession` / `RealtimeSession`)
Managing a continuous, real-time connection requires tracking state, context, and handling interruptions.

*   **`AgentSession` (`voice/agent_session.ts`):** Acts as the central hub for a single conversation. It manages:
    *   The user's current state (listening, speaking).
    *   The agent's current state (initializing, listening, speaking, thinking).
    *   Error recovery and reconnection logic (tracking `llmErrorCounts`, `ttsErrorCounts`).
*   **Context Management (`ChatContext`):** LiveKit uses a `ChatContext` object to maintain the history of the conversation (messages, tool calls, and results). This is essential for the LLM to understand the ongoing dialogue.
*   **Turn Detection and Interruption Handling:** LiveKit includes logic to detect when a user starts or stops speaking (`turnDetection`), and sophisticated handling for when a user interrupts the agent (`allowInterruptions`), which involves cancelling the current TTS/LLM generation and updating the context.

**What we can use:** The state machine design for the session, the mechanism for managing conversation history, and the patterns for handling real-time interruptions gracefully.

## 3. Streaming and Asynchronous Generation (`generation.ts`)
Real-time agents cannot wait for a full response before acting. They must stream data continuously.

*   **ReadableStreams:** LiveKit extensively uses Web Streams API (`ReadableStream`) to pipe data between components.
*   **Parallel Tool Execution:** The framework handles streaming tool calls from the LLM, executing them, and feeding the results back into the context asynchronously.
*   **Time-to-First-Byte (TTFB) Tracking:** Performance is critical, and LiveKit tracks metrics like TTFB to monitor latency in the generation process.

**What we can use:** The heavy reliance on asynchronous streaming for both text and audio, allowing the agent to begin speaking as soon as the first few words are generated by the LLM and synthesized by the TTS.

## 4. Hardware/Network I/O abstraction (`voice/io.ts` & `worker.ts`)
LiveKit separates the agent logic from the underlying network transport.

*   **`AgentInput` / `AgentOutput`:** Abstractions for receiving audio/video from the user and sending audio/video back.
*   **Workers (`worker.ts`):** LiveKit uses a worker model to scale agents. A worker process connects to the LiveKit server, listens for new job assignments, and spins up new `AgentSession` instances. This handles CPU monitoring and load balancing.

**What we can use:** The abstraction of I/O allows us to develop the agent logic independently of whether it's connecting via WebRTC (LiveKit), WebSockets, or another transport layer. The worker pattern is useful for scaling.

## 5. Tool Calling and LLM Integration (`llm/index.ts`)
LiveKit provides a unified interface for interacting with different LLM providers (OpenAI, Anthropic, etc.).

*   **`ToolContext`:** A registry of functions the LLM can call.
*   **Zod Integration:** Using Zod schemas to define expected structured output from the LLM.
*   **Realtime Model Support (`RealtimeModel`):** Specific support for models that can natively handle audio input/output (like OpenAI's Realtime API), bypassing the separate STT/TTS nodes.

**What we can use:** The standardized way of defining tools, serializing them for the LLM, and parsing the LLM's tool calls to execute local functions.

## Summary of Actionable References
If we build our own real-time AI agent, we should closely examine and adapt the following from LiveKit Agents-JS:
1.  **The Stream Pipeline:** Connecting `ReadableStream`s from Audio Input -> STT -> LLM -> TTS -> Audio Output.
2.  **Interruption Logic:** How `AgentActivity` cancels ongoing LLM/TTS generation when the VAD (Voice Activity Detector) triggers a new user speech event.
3.  **Chat Context Management:** How tool calls, tool results, and standard messages are interleaved in the context history.
4.  **The unified LLM/STT/TTS interfaces:** Allowing easy swapping of providers (e.g., swapping OpenAI TTS for Cartesia or ElevenLabs) without changing the core agent logic.
